# Ключевые ограничения и упрощения

Явная фиксация ограничений и упрощений реализации. Соответствует критерию B7-9.

---

## Запуск и доступ

- **Первый вход:** Для входа в систему нужен хотя бы один пользователь с ролью Admin. Он создаётся при старте только если заданы переменные окружения `ADMIN_EMAIL` и `ADMIN_PASSWORD`. Иначе первого админа нужно создать вручную (миграция/скрипт/прямая вставка в БД).
- **Регистрация:** `POST /auth/register` доступен только администратору. Обычные пользователи не могут регистрироваться сами.

---

## Ревью и апрувер-группы

- **Fallback при отсутствии апрувер-группы:** Если у Experimenter не задана апрувер-группа (approver_ids / min_approvals_required), поведение при ревью определяется текущей доменной логикой (например, одобрение только от Admin или порог 1). Конкретный fallback описан в коде (`can_be_launched` / проверка одобрений) и должен совпадать с описанием в runbook.
- **Апрувер-группа при создании пользователя:** Указывается опционально только для роли Experimenter; для остальных ролей поля `approver_ids` и `min_approvals_required` игнорируются.

---

## События и метрики

- **Окно атрибуции:** События с большой задержкой (например, старше 7 дней относительно более нового события) могут не учитываться в атрибуции в зависимости от реализации — см. ТЗ 4.4.2.
- **Хранение решений:** Нет автоматического TTL для записей решений (decisions); при очень большом объёме может потребоваться партиционирование или архивация.
- **Redis и метрики:** Временные агрегаты для guardrails и скользящих окон хранятся в Redis. При недоступности Redis проверка guardrails и расчёт метрик за окно могут быть невозможны до восстановления.

## Уведомления (допфича)

- **Каналы:** Реализованы доставки в Telegram и Slack через **Incoming Webhook** (HTTP POST в webhook URL). Отдельный Slack-бот с Bolt/Events API не используется; для отправки в канал достаточно создать Incoming Webhook в настройках Slack-приложения. URL webhook хранится в конфигурации канала и в ответах API маскируется (не логируется в открытом виде).
- **Доставка:** Асинхронная через Celery. При неработающем воркере сообщения накапливаются в очереди; повторные попытки и ограничение частоты заданы конфигом.
- **Дедупликация с учётом времени:** Один и тот же `event_id` обрабатывается и доставляется не более одного раза (таблица `notification_events`). Для срабатываний guardrail в `event_id` входит минутный тайм-слот (1 мин), поэтому повторное срабатывание того же guardrail в другую минуту порождает новое уведомление; в пределах одной минуты — одно уведомление. Для событий жизненного цикла эксперимента используется версия (один раз на переход).
- **Шаблоны:** Формат сообщений задаётся в коде (NotificationMessageFormatter); кастомизация шаблонов через админку не реализована.

## Learnings Library (допфича)

- **OpenSearch:** Поиск похожих экспериментов и индексация learnings зависят от OpenSearch. При недоступности OpenSearch приложение стартует, но поиск похожих возвращает пустой результат; обновление learning может падать или пропускаться в зависимости от обработки ошибок.
- **Индекс:** Имя индекса задаётся переменной `OPENSEARCH_INDEX`. Миграции схемы индекса (поля, маппинг) при изменении модели выполняются отдельно от основного приложения.
- **Полнота:** Заполнение learning при завершении эксперимента может быть частично обязательным (зависит от политики); детали — в описании допфичи и в матрице соответствия.

## Масштабирование и производительность (B9-6, B9-7)

- **Нагрузка:** Приложение не тестировалось под высокой нагрузкой; при росте трафика может потребоваться пул воркеров, шардирование Redis и настройка пула БД.
- **Индексы и запросы:** Критичные запросы (решения по субъекту, события по эксперименту/варианту, отчёты) опираются на индексы БД и лимиты выборки там, где они заданы. Явное описание индексов и мер эффективности приведено в runbook или в комментариях к репозиториям при необходимости.

---

## Демо и проверка

- В e2e-тестах админ создаётся при старте за счёт `ADMIN_EMAIL`/`ADMIN_PASSWORD` в окружении; регистрация через API в тестах не используется.
- Ограничения, влияющие на сценарии демо (например, обязательность OpenSearch для поиска похожих экспериментов), отражены в `docs/demo_scenarios_ru.md` и в матрице соответствия.
