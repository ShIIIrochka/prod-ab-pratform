# Ключевые архитектурные решения

Документ фиксирует основные архитектурные решения проекта и их последствия (риски/ограничения). Соответствует критерию B7-4.

---

## 1. Единый HTTP-API и фоновые задачи в одном процессе

**Решение:** Основной API (FastAPI) и фоновый цикл проверки guardrails (GuardrailCheckerWorker) работают в одном процессе приложения. Очередь уведомлений вынесена в Celery (отдельный процесс).

**Обоснование:** Упрощение развёртывания и отсутствие необходимости в отдельном планировщике для guardrails. Критичный путь decide → events → report не зависит от Celery.

**Риски/ограничения:** При высокой нагрузке цикл guardrails конкурирует с HTTP-запросами за один event loop; интервал проверки задаётся конфигом (`GUARDRAIL_CHECK_INTERVAL_SECONDS`). При необходимости можно вынести воркер в отдельный процесс без изменения контрактов.

---

## 2. Регистрация пользователей только для администратора

**Решение:** Эндпоинт `POST /auth/register` доступен только роли Admin. Первый администратор создаётся при старте приложения, если в окружении заданы `ADMIN_EMAIL` и `ADMIN_PASSWORD`.

**Обоснование:** Исключение саморегистрации и контроль состава пользователей платформы.

**Риски/ограничения:** Без заданных переменных окружения первый вход в систему невозможен до ручного создания админа через БД или отдельный скрипт. В e2e-тестах админ создаётся при старте через эти переменные.

---

## 3. Апрувер-группа для Experimenter (опционально)

**Решение:** При создании пользователя с ролью Experimenter можно опционально указать `approver_ids` и `min_approvals_required`. Если не указать — апрувер-группа не задаётся; поведение при ревью в этом случае определяется доменной логикой (например, fallback на единственного одобрения или только Admin).

**Обоснование:** Гибкость: можно настроить ревью под команду или не задавать группу для упрощённых сценариев.

**Риски/ограничения:** Fallback-поведение при отсутствии группы должно быть явно описано в документации и воспроизводимо (см. ТЗ 0.3). В коде проверяется наличие апрувера в списке владельца или роль Admin.

---

## 4. Хранение решений (decisions) и атрибуция событий

**Решение:** Решение о выдаче варианта (decision_id) сохраняется в БД и связывается с субъектом, флагом и экспериментом. События принимаются с указанием `decision_id`; атрибуция конверсий к экспозиции выполняется по правилам каталога событий и хранится в репозитории событий.

**Обоснование:** Обеспечение воспроизводимости и корректной аналитики при задержках и переупорядочивании событий.

**Риски/ограничения:** При очень большом объёме решений может потребоваться партиционирование или TTL; текущая реализация предполагает хранение без автоматической очистки.

---

## 5. Уведомления через Celery и доменные события

**Решение:** Доменные события (например, GuardrailTriggered, ExperimentLaunched) публикуются через DomainEventPublisher; диспетчер уведомлений подписывается на них и ставит задачи в Celery. Доставка — Telegram, Slack по конфигурируемым каналам.

**Обоснование:** Развязка бизнес-логики и каналов доставки; возможность добавлять каналы без изменения use case.

**Риски/ограничения:** Доставка асинхронная; при падении воркера уведомления накапливаются в очереди. Rate limiting и дедупликация реализованы на уровне правил и хранилища доставок.

---

## 6. Learnings Library на OpenSearch

**Решение:** Поиск похожих экспериментов и обновление learning-записей реализованы через OpenSearch (индекс learnings). При недоступности OpenSearch приложение стартует, но поиск похожих возвращает пустой результат.

**Обоснование:** Полнотекстовый и структурированный поиск без нагрузки на основную БД.

**Риски/ограничения:** Зависимость от ещё одного сервиса; необходимость реиндексации при смене схемы. Ограничения и упрощения описаны в `docs/architecture/limitations_ru.md`.

---

## 7. Метрики и guardrails на агрегатах в Redis

**Решение:** Временные агрегаты для расчёта метрик по окну наблюдения и для отчётов хранятся/читаются через MetricAggregatorPort (реализация — Redis). Отчёты по эксперименту строятся по событиям из PostgreSQL с учётом атрибуции.

**Обоснование:** Быстрый доступ к скользящим окнам для guardrails и возможность масштабирования без блокировок основной БД.

**Риски/ограничения:** При падении Redis метрики за окно теряются; постоянная история — в событиях в БД. Подробнее в разделе про ограничения.
